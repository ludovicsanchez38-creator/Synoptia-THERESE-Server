## Problèmes critiques (action immédiate)

1. **Fuite cross-tenant** : tasks.py, crm.py et ~19 autres routers n'utilisent pas scope_query. Un utilisateur authentifié accède aux données de tous les utilisateurs.
2. **Legal.tsx ment** : affirme "conversations chiffrées Fernet" - faux, seules les clés API sont chiffrées.
3. **Seed admin/admin** : mot de passe hardcodé dans seed.py, déployé en production via docker-compose.
4. **Secrets par défaut** : jwt_secret et secret_key = "changeme..." en absence de .env.

## Plan d'action

### Semaine 1 (Quick Wins)
- Corriger Legal.tsx (retirer mention chiffrement conversations)
- Bloquer démarrage si jwt_secret contient "changeme" en production
- Seed : générer mot de passe aléatoire + l'afficher une fois
- /health/services : retourner 503 + exiger auth
- Ajouter jitter au retry backoff

### Semaine 2-3 (Structurel)
- Audit systématique des 28 routers : ajouter CurrentUser + scope_query
- Étendre la CI à tous les tests backend
- Pagination + filtre user_id sur /data/export
- Configurer monitoring externe (UptimeKuma)
- Activer python-json-logger dans main.py

### Mois 2+ (Évolutions)
- DPO configurable par instance + mentions légales dynamiques
- Mot de passe oublié + reset par email
- Vue Kanban CRM
- Chiffrement Fernet des conversations (aligner avec la déclaration)
- Tests composants frontend (30%+ couverture)

## Résumé exécutif

Thérèse Server progresse de 59.7 à 64.6/100 grâce à la CI/CD, au Dockerfile multi-stage, au circuit breaker LLM, aux tests Vitest et au nettoyage ruff. Cependant, l'audit révèle une **fuite cross-tenant critique** sur ~19 routers (les données d'un utilisateur sont accessibles par tous les autres) et une **fausse déclaration de chiffrement** dans les mentions légales. Le DevOps reste le point faible (52/100) malgré la CI existante, principalement par manque de monitoring et de couverture de tests en CI. La priorité absolue est l'isolation multi-tenant avant tout déploiement collectivité.
